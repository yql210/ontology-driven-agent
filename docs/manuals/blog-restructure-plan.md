# 博客重构方案

## 诊断

当前 10 篇的问题：

| 问题 | 后果 |
|------|------|
| 按项目时间线推进 | 读起来像开发日志，不是技术博客 |
| 跨篇依赖链条长 | "上篇我们做了 X…" — 独立读者直接关 |
| 项目名刷屏 | "LayerKG" 每篇出现 20+ 次，自我推销感强 |
| 配置命令/内部术语 | CLI 示例、内部路径名 — 对读者零价值 |
| 缺少独立论点 | 每篇是"我做了 X"，不是"这个问题应该怎么思考" |

## 重构原则

1. **每一篇单独成立** — 不需要读前一篇，独立可分享
2. **项目是案例，不是主角** — 正文只说 1-2 次项目名，在文末"关于本文案例"块中说明
3. **标题是对外标题** — 可以被单独转发到 V2EX/掘金/知乎/朋友圈
4. **开篇是问题** — 前 3 段内必须有：这个问题为什么值得关心
5. **必须有对比表和面试追问** — 每篇标配

## 新 10 篇结构

### 1. 《代码知识图谱的 Schema 该怎么设计？》
**来源**：砍掉旧 1（Schema 巡礼）+ 旧 7（论文→落地），取核心决策
**核心论点**：好的代码 KG Schema = 可演化 + 可查询 + 可增量，不是 ER 图画完就完
**卖点**：从 6 实体→11 实体的演化故事 + "parameters 属性名冲突 Neo4j driver"的坑

### 2. 《tree-sitter 多语言解析的 6 个坑》
**来源**：旧 2，重写为纯陷阱文章
**核心论点**：tree-sitter 是当前最好的多语言解析器，但工程落地每个语言都有暗坑
**卖点**：Python bytes/text 陷阱、Java 泛型继承、0-indexed 行号、Query API 废弃 — 每个坑都有代码对比

### 3. 《1 行代码改动能影响 319 个节点：增量更新的 BFS 传播算法》
**来源**：旧 3
**核心论点**：Git diff → 影响传播不是简单的依赖图遍历，需要双向 BFS + 衰减权重
**卖点**：1 file → 319 nodes 的真实数据，BFS 实现（不是理论），与其他工具对比

### 4. 《91.4% 的 Agent 准确率不是调参调出来的》
**来源**：旧 4
**核心论点**：Agent 准确率提升靠 eval 方法论 + 工具设计，不是 prompt 迭代
**卖点**：35 题 3 级评测体系，L1→L3 难度梯度，工具匹配率 92.9%

### 5. 《Agent 需要自己的心跳：事件驱动知识管理引擎》
**来源**：旧 5
**核心论点**：AI Agent 如果不是响应式的而是事件驱动的，架构会完全不同
**卖点**：EventBus asyncio.Queue 模式、3 个 Handler 管道、GitWatcher polling strategy

### 6. 《AI 辅助开发时代，TDD 更重要了》
**来源**：旧 6
**核心论点**：AI 写代码越快，TDD 的 safety net 价值越大 — 不是反AI，是让AI更安全
**卖点**：从 0→1038→1457 的三阶段测试策略演进

### 7. 《为什么 95 分的架构被我亲手推翻》
**来源**：旧 8
**核心论点**：AI Code Review 给高分不代表架构正确 — Claude Code 95分 vs 实际业务匹配度的冲突
**卖点**：AI 审查看「设计完整度」，不看「业务适配度」；过度设计的识别标准

### 8. 《SAGA 模式不只属于微服务——单进程 Agent 的容错编排》
**来源**：旧 9
**核心论点**：SAGA/熔断/重试这些分布式模式在单进程 Agent 编排中同样有效
**卖点**：5 态状态机 + CircuitBreaker 三态 + 补偿 vs 回滚的决策

### 9. 《给 AI Agent 造一个「语义笼子」》 🆕
**来源**：Phase 5 约束框架
**核心论点**：与其事后审计 Agent 越权，不如用本体定义把行为约束在语义边界内
**卖点**：三层约束架构（自动推导→YAML覆盖→运行时动态）、图遍历约束链、白名单豁免模式

### 10. 《Agent 操作数据库时，什么时候必须等人类批准？》 🆕
**来源**：Phase 5 审批系统
**核心论点**：AI Agent 的审批不是开关（开/关），而是分级策略链
**卖点**：function/action 双级审批、令牌一次性+TTL、策略链可插拔、审计追踪

