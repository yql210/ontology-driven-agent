## 2. 设计方案

### 2.1 修复 IMPORTS 关系（P0）

**根因**：`_extract_import()` 取顶层模块名 `split(".")[0]`，如 `from layerkg.schema import ...` → target="layerkg"。而 resolve 的 `name_to_id` 用 `e.name`（文件 stem="schema"），所以项目内 import 也被过滤。

**方案**：分两步解决——先改提取逻辑让项目内 import 能 resolve，再改 resolve 让外部 import 也保留。

#### 步骤一：修改提取逻辑

**`_extract_import()`（处理 `import X.Y.Z`）**：
- target_name 改为**最后一段**（`Z`），因为 all_entities 中 module 实体的 name 就是文件 stem
- 例：`import os.path` → target="path"；`import layerkg.schema` → target="schema" ✅

```python
def _extract_import(self, node, source, file_path, entities, relations, module_name):
    for child in node.children:
        if child.type == "dotted_name" or child.type == "aliased_import":
            if child.type == "aliased_import":
                name_part = child.child_by_field_name("name")
                if name_part:
                    import_name = name_part.text.decode()
                else:
                    continue
            else:
                import_name = child.text.decode()
            
            last_segment = import_name.split(".")[-1]  # 改为取最后一段
            
            relation = ExtractedRelation(
                source_name=module_name,
                source_type="module",
                target_name=last_segment,
                target_type="module",
                relation_type="imports",
                file_path=file_path,
            )
            relations.append(relation)
```

**`_extract_import_from()`（处理 `from X.Y import A, B`）**：
- 模块关系：target 取**最后一段**（`Y`），不是顶层
- 具名导入：额外提取 `A`, `B`，target_type 暂设 "module"（resolve 按实际匹配）

```python
def _extract_import_from(self, node, source, file_path, entities, relations, module_name):
    module_node = node.child_by_field_name("module_name")
    if module_node is None:
        return
    
    import_module = module_node.text.decode()  # "layerkg.schema"
    last_segment = import_module.split(".")[-1]  # "schema"
    
    # 模块级 import
    relations.append(ExtractedRelation(
        source_name=module_name,
        source_type="module",
        target_name=last_segment,
        target_type="module",
        relation_type="imports",
        file_path=file_path,
    ))
    
    # 提取 from X import A, B 中的具体名称
    for child in node.children:
        if child.type in ("dotted_name", "identifier", "aliased_import"):
            if child == module_node:
                continue
            if child.type == "aliased_import":
                name_part = child.child_by_field_name("name")
                if name_part:
                    imported_name = name_part.text.decode()
                else:
                    continue
            else:
                imported_name = child.text.decode()
            
            relations.append(ExtractedRelation(
                source_name=module_name,
                source_type="module",
                target_name=imported_name,
                target_type="module",
                relation_type="imports",
                file_path=file_path,
            ))
```

#### 步骤二：修改 resolve 策略（保持向后兼容）

**不修改 `resolve()` 返回类型**。新增 `resolve_with_unresolved()` 方法：

```python
# extractor/relation.py
def resolve(self, all_entities: list[CodeEntity]) -> list[Relation]:
    """原有接口不变。"""
    resolved, _ = self.resolve_with_unresolved(all_entities)
    return resolved

def resolve_with_unresolved(self, all_entities: list[CodeEntity]) -> tuple[list[Relation], list[ExtractedRelation]]:
    """解析关系，同时返回未 resolve 的外部依赖。"""
    name_to_id = self._build_name_map(all_entities)
    resolved = []
    unresolved = []
    
    for rel in self._relations:
        source_id = name_to_id.get(rel.source_name)
        target_id = name_to_id.get(rel.target_name)
        if source_id and target_id:
            resolved.append(Relation(source_id=source_id, target_id=target_id, relation_type=rel.relation_type))
        elif rel.relation_type == "imports" and source_id:
            unresolved.append(rel)
    
    return resolved, unresolved
```

### 2.2 新增 CALLS 关系提取（P0）

**方案**：在 `_extract_function()` 中遍历函数体 AST，提取 call 调用。resolve 时使用**同文件优先**策略解决同名歧义。

tree-sitter Python 中 call 表达式结构：
```
call
├── identifier (函数名) 或 attribute (obj.method)
└── argument_list
```

**新增方法** `_extract_calls()`：

```python
# Python 内置函数名黑名单（不提取 CALLS 关系）
_BUILTIN_NAMES = frozenset({
    "print", "len", "str", "int", "float", "bool", "list", "dict", "set",
    "tuple", "range", "enumerate", "zip", "map", "filter", "sorted",
    "isinstance", "issubclass", "hasattr", "getattr", "setattr", "delattr",
    "type", "super", "property", "classmethod", "staticmethod",
    "abs", "all", "any", "bin", "chr", "dir", "hex", "id", "input",
    "max", "min", "next", "oct", "open", "ord", "pow", "repr", "round",
    "sum", "vars", "format", "hash", "help", "iter", "reversed",
    "Exception", "ValueError", "TypeError", "RuntimeError", "KeyError",
    "IndexError", "AttributeError", "ImportError", "FileNotFoundError",
    "NotImplementedError", "StopIteration", "OSError",
})

def _extract_calls(self, func_node, source, file_path, relations, module_name, caller_name):
    """提取函数体中的 call 关系。"""
    call_queue = list(func_node.children)
    while call_queue:
        child = call_queue.pop(0)
        if child.type == "call":
            callee_node = child.child_by_field_name("function")
            if callee_node:
                callee_name = callee_node.text.decode()
                # 处理 obj.method / self.method / Class.method 形式
                if "." in callee_name:
                    parts = callee_name.split(".")
                    # 统一取最后一段（self.method → method, Module.Class.method → method）
                    callee_name = parts[-1]
                
                # 过滤内置函数
                if callee_name in _BUILTIN_NAMES:
                    continue
                # 过滤太短的名称（可能是变量）
                if len(callee_name) < 3:
                    continue
                
                relations.append(ExtractedRelation(
                    source_name=caller_name,
                    source_type="function",
                    target_name=callee_name,
                    target_type="function",
                    relation_type="calls",
                    file_path=file_path,
                ))
            # 不递归 call 的子节点（避免重复提取 argument_list 内的 call）
        else:
            # 递归其他子节点（捕获 if/for/with 等块内的 call）
            call_queue.extend(child.children)
```

在 `_extract_function()` 末尾调用：
```python
self._extract_calls(node, source, file_path, relations, module_name, full_name)
```

**resolve 策略——同文件优先**：

当前 `_build_name_map` 返回 `{e.name: e.id}`，同名实体会互相覆盖。修改为支持多值映射：

```python
# extractor/relation.py — _build_name_map 改为多值
@staticmethod
def _build_name_map(entities: list[CodeEntity]) -> dict[str, list[str]]:
    """构建名称到 ID 列表的映射（支持同名实体）。"""
    mapping: dict[str, list[str]] = {}
    for e in entities:
        mapping.setdefault(e.name, []).append(e.id)
    return mapping
```

resolve 时根据关系类型使用不同策略：

```python
def _resolve_relation(self, rel: ExtractedRelation, name_to_ids: dict[str, list[str]], 
                      entities_by_id: dict[str, CodeEntity]) -> tuple[str, str] | None:
    """解析单条关系，返回 (source_id, target_id) 或 None。"""
    source_ids = name_to_ids.get(rel.source_name, [])
    target_ids = name_to_ids.get(rel.target_name, [])
    if not source_ids or not target_ids:
        return None
    
    # 对于 CALLS 关系，优先匹配同文件内的实体
    if rel.relation_type == "calls" and len(target_ids) > 1:
        for tid in target_ids:
            entity = entities_by_id.get(tid)
            if entity and entity.file_path == rel.file_path:
                return source_ids[0], tid
    # 对于 IMPORTS 关系，优先匹配 module 类型
    elif rel.relation_type == "imports" and len(target_ids) > 1:
        for tid in target_ids:
            entity = entities_by_id.get(tid)
            if entity and entity.entity_type == "module":
                return source_ids[0], tid
    
    return source_ids[0], target_ids[0]
```

**向后兼容**：resolve() 方法内部改用新的多值映射+策略匹配，但返回值类型不变（`list[Relation]`）。现有调用方无需修改。

```python
# 完整的 resolve() 实现
def resolve(self, all_entities: list[CodeEntity]) -> list[Relation]:
    """原有接口不变。"""
    name_to_ids = self._build_name_map(all_entities)
    entities_by_id = {e.id: e for e in all_entities}
    resolved = []
    
    for rel in self._relations:
        result = self._resolve_relation(rel, name_to_ids, entities_by_id)
        if result:
            source_id, target_id = result
            resolved.append(Relation(source_id=source_id, target_id=target_id, relation_type=rel.relation_type))
    
    return resolved

def resolve_with_unresolved(self, all_entities: list[CodeEntity]) -> tuple[list[Relation], list[ExtractedRelation]]:
    """解析关系，同时返回未 resolve 的外部依赖。"""
    name_to_ids = self._build_name_map(all_entities)
    entities_by_id = {e.id: e for e in all_entities}
    resolved = []
    unresolved = []
    
    for rel in self._relations:
        result = self._resolve_relation(rel, name_to_ids, entities_by_id)
        if result:
            source_id, target_id = result
            resolved.append(Relation(source_id=source_id, target_id=target_id, relation_type=rel.relation_type))
        elif rel.relation_type == "imports" and name_to_ids.get(rel.source_name):
            unresolved.append(rel)
    
    return resolved, unresolved
```

### 2.3 新增 IMPLEMENTS 关系（P1）

**方案**：在 `_extract_class()` 中区分"继承自抽象基类"和"继承自普通类"。

由于纯 AST 分析无法确定父类是否为抽象基类（需要类型信息），采用启发式规则：
1. 父类名以 `ABC`/`Protocol`/`Interface` 结尾 → `IMPLEMENTS`
2. 父类名以 `Base`/`Abstract` 开头 → `IMPLEMENTS`
3. 被继承的类在项目内且被 `abc.ABCMeta` 元类标记（检查类体是否有抽象方法） → `IMPLEMENTS`
4. 其余 → `EXTENDS`（保持现有行为）

```python
# 在 _extract_class() 的 superclasses 遍历中
if self._is_interface_like(parent_name, node):
    relation_type = "implements"
else:
    relation_type = "extends"
```

**简化方案**（推荐）：由于当前项目 EXTENDS 只有 9 条，且都是明确的类继承，暂不引入 IMPLEMENTS 区分。保持 EXTENDS 覆盖所有继承关系。IMPLEMENTS=0 不会阻塞 Phase 3。

**决策**：**暂不实施**。理由：
- 当前项目无抽象基类/接口定义（全部是具体类继承）
- AST 层面无法区分继承和实现（需类型信息）
- Agent 可以通过 EXTENDS 遍历整个继承树

### 2.4 提升 SEMANTIC_IMPACT 数量（P1）

**根因分析**：当前 prompt 中 `semantic_impact` 和 `derived_from` 混在一个 prompt 里，但 prompt 强调 "Step 2: Concepts (MOST IMPORTANT)"，导致 LLM 偏重 derived_from 而忽略 semantic_impact。

**方案**：修改 `_build_prompt()` 的权重和指导：

1. 将 Step 1 (semantic_impact) 和 Step 2 (concepts) 平等对待，去掉 "(MOST IMPORTANT)" 标记
2. 增加 semantic_impact 的示例和引导：
   ```
   Common semantic impact patterns:
   - Data flow: parse() → validate() → transform() → store()
   - Config dependency: functions that read the same config object
   - Error propagation: function A catches errors from function B
   - Side effects: function A modifies state that function B reads
   ```
3. 在 Rules 中增加：
   ```
   - You MUST identify both semantic_impact AND derived_from relations
   - Aim for roughly equal numbers of each relation type
   ```

### 2.5 减少 DocEntity 孤立（P2）

**根因**：601/714 (84%) DocEntity 孤立。`_link_docs_to_code()` 匹配策略：
1. 路径精确匹配：doc.content 包含代码文件路径 → 太严格
2. 代码块标识符匹配：只看 ```python``` 代码块 → 大量纯文本 doc 被遗漏

**方案**：增强 `_link_docs_to_code()` 的匹配策略：

1. **文件名匹配**（新增）：doc.content 包含 `module_name.py` 形式 → 匹配该文件的所有实体
2. **类名/函数名全文匹配**（新增）：doc.content 中出现的类名/函数名 → 直接匹配（不再限于代码块）
3. **标题匹配**（新增）：DocEntity 的 name/title 中包含类名/函数名 → 匹配
4. **同目录匹配**（新增）：doc.file_path 和 entity.file_path 在同一目录 → 弱关联

```python
# 新增匹配策略（按优先级）
strategies = [
    self._match_by_path,        # 现有：路径精确匹配
    self._match_by_filename,    # 新增：文件名.py
    self._match_by_title,       # 新增：doc.title/name 包含实体名
    self._match_by_content_name, # 新增：全文中出现实体名
    self._match_by_code_block,  # 现有：代码块标识符
]
```

### 2.6 补全 ModuleEntity 属性（P2）

**根因**：`ModuleClustering.save_modules()` 只写 `name`，不写 `description` 和 `size`。

**方案**：在 `save_modules()` 中：
1. `size` = cluster 中 CodeEntity 的数量
2. `description` = 聚类中最具代表性的实体描述（取 class 类型的实体名列表，或 function 数量描述）

```python
# module_clustering.py — save_modules() 改为：
def save_modules(self, clusters: list[ModuleCluster], all_entities: list[CodeEntity] = None) -> None:
    # ModuleCluster 只有 entity_ids，需从 all_entities 中查找
    entity_by_id = {e.id: e for e in (all_entities or [])}
    
    for cluster in clusters:
        entities = [entity_by_id[eid] for eid in cluster.entity_ids if eid in entity_by_id]
        size = len(cluster.entity_ids)
        class_names = [e.name for e in entities if e.entity_type == "class"]
        func_count = sum(1 for e in entities if e.entity_type == "function")
        
        if class_names:
            desc = f"Module containing {', '.join(class_names[:3])}"
            if len(class_names) > 3:
                desc += f" and {len(class_names)-3} more classes"
            desc += f", {func_count} functions"
        else:
            desc = f"Module with {func_count} functions"
        
        self._graph_store.merge_node("ModuleEntity", {
            "id": cluster.module.id,
            "name": cluster.module.name,
            "description": desc,
            "size": size,
        })
```

### 2.7 填充 ConceptEntity.concept_type（P3）

**根因验证**：经 Claude Code 审核确认，`_process_semantic_relations()` 中创建 ConceptEntity 的代码（builder.py ~L873）已正确传递 `entity_type=target_type`。实际数据库中 `concept_type=None` 是因为**查询属性名不对**——Schema 中该字段叫 `entity_type`（不是 `concept_type`），但 Neo4j 审计查询使用了 `n.concept_type`，Neo4j 报 "property does not exist" 警告。

**验证**：
```cypher
MATCH (n:ConceptEntity) RETURN n.entity_type AS et, count(n) LIMIT 5
```

**方案**：先验证 `entity_type` 字段实际值。如果确认是审计查询错误，则 2.7 不需要代码修改，只需修正审计脚本。

## 6. 风险与缓解

| 风险 | 影响 | 缓解 |
|------|------|------|
| CALLS 提取产生大量噪声（如 built-in 函数名） | 关系膨胀 | `_BUILTIN_NAMES` 黑名单 + 长度过滤 (<3) + resolve 只匹配 all_entities 中已有实体 |
| 同名函数导致 CALLS 指向错误目标 | 数据错误 | resolve 同文件优先匹配（`file_path == rel.file_path`） |
| IMPORTS 外部模块创建大量虚拟节点 | Neo4j 膨胀 | 限制只创建项目直接依赖（不递归传递依赖），外部节点 file_path="__external__" |
| `_build_name_map` 改为多值影响现有 CONTAINS/EXTENDS | 现有测试失败 | CONTAINS/EXTENDS 的 source 是 module/class，几乎不重名；resolve 内部自动适配 |
| prompt 修改可能降低 derived_from 产出 | 概念减少 | 保留 derived_from 权重提示，只是平衡两者；可回滚 |
| DocEntity 全文名匹配产生误报 | 虚假 DESCRIBES | 名字长度 >5 字符才匹配；排除常见词（get, set, run, add 等） |
| 全量重建耗时 146 分钟 | 验证周期长 | 每步先用小规模测试验证，最后一步全量构建 |
